## 二、关于向量数据库和标签的讨论

你的直觉是对的——对于 MVP，**不一定需要向量数据库**。让我解释清楚：

### 标签系统（MVP 推荐）

给文章和词条打标签，可以实现基本的关联和检索：
- 手动标签 + AI 自动建议标签
- 标签之间可以有关联（"动态规划"属于"算法"）
- 基于标签的简单推荐："同标签文章"

这完全够用，且实现简单，不需要引入向量数据库。

### 向量数据库是做什么的

它解决的是**语义相似度**问题——不是匹配关键词，而是理解意思。比如：
- 搜索"怎么优化时间复杂度"，能找到讲"复杂度分析"的文章，即使文章里没有"优化"这个词
- 读者对文章提问时，找到文章中最相关的段落

**现阶段判断：你不需要**

理由：
1. 你的文章对话限定在当前文章内，不需要跨文章检索
2. 词条关联可以靠 AI 在发布时主动分析 + 标签系统，不需要实时语义搜索
3. 引入向量数据库会增加架构复杂度和成本

**将来什么时候需要**：如果你想做"全站语义搜索"或"跨文章的智能问答"，那时再上。架构上可以预留扩展点，但不现在实现。

## 三、关于"数据直接文本形式存储"

你提到"不需要数据库"，我理解你的意思是想以 Markdown 文件为主要存储。这是一个需要仔细考虑的架构决策：

### 纯文件存储的问题

| 场景 | 纯 Markdown 文件 | 数据库 + 文件 |
|------|------------------|--------------|
| 查"某标签下所有文章" | 遍历所有文件读 frontmatter | 一条 SQL 查询 |
| 查"引用了词条X的所有文章" | 每次全量扫描 | 关联表查询 |

发布流程有两种路径：
1. **直接发布**：草稿 → 已发布（跳过审查）
2. **审查后发布**：草稿 → 审查中 → 待修改（可选循环） → 已发布

#### 3.1.8 AI 审查功能

博主提交文章时可以选择"直接发布"或"提交审查"。

审查报告包含以下板块：
- **事实性错误**：知识性、逻辑性错误
- **拼写与语法**：错别字、语法问题
- **结构与可读性**：段落组织、表达清晰度
- **其他建议**：不属于以上类别的建议

每个问题条目包含：
- 严重程度：`error` / `warning` / `suggestion`
- 位置：行号范围 + 原文摘录
- 问题简述
- 修改建议

审查报告仅供博主内部使用，不对外公开，不进入文章版本历史。

#### 3.1.9 评论功能

- **必须登录**：仅登录用户可发表评论
- **频率限制**：同一用户对同一文章在 60 秒内只能发表一条评论
- **AI 初审**：Miniese 自动检测评论内容，若识别为不当内容（辱骂、广告、垃圾信息等），则自动隐藏该评论并通知博主
- 博主看到评论后可直接回复，无需事前审核

### 3.2 知识库（Wiki）

#### 3.2.1 词条定义

词条是知识库的基本单位，用于解释概念、定理、术语等。词条之间可通过链接关联。

#### 3.2.2 词条属性

- **主名称**：词条的标准名称（英文）
- **别名**：字符串数组，包含中文译名、简称、常见变体等
- **内容**：支持 Markdown / Notesaw 语法
- **标签**：与文章共享标签体系
- **可见性**：`accessGroups` 数组
- **参考文献**：规范化的引用列表

#### 3.2.3 多语言策略

中英文词条是**独立的两个词条**，不自动翻译，仅通过链接关联：
- 中文词条 URL：`/zh/wiki/词条名`
- 英文词条 URL：`/en/wiki/term-name`
- 页面顶部语言切换按钮会跳转到对应语言的词条（如不存在则显示 404 或创建提示）

AI 助手撰写词条时，分别撰写中文和英文版本，而不是翻译。

#### 3.2.4 双向链接

- **文章 → 词条**：渲染文章时，自动检测文本中出现的词条主名称或别名，添加 hover 预览链接
  - hover 显示：定义摘要 + 跳转链接
- **词条 → 文章**：词条页面底部自动展示"引用了此词条的文章"列表（反向链接）

#### 3.2.5 词条来源与标注

| 来源 | 说明 | 标注 |
|------|------|------|
| 人类撰写 | 博主手动创建或编辑 | 无额外标注 |
| AI 生成 | 助手自动发现新概念并生成词条草稿 | 标注"AI生成"，并显示是否经人工审查 |

所有 AI 生成内容必须标注引用来源（外部维基、教材、网页等）。

#### 3.2.6 词条申请

读者可提交词条申请（提议创建新词条），内容包含：词条名称、简要说明、补充材料等。申请进入管理员仪表盘，由博主审批后决定是否创建及由谁创建（人工或 AI）。

关于移动端适配，方案一的悬浮按钮在移动端会缩小并移到右下角，点击后抽屉从底部弹出，这是成熟的设计模式。

**区域顺序的考量**：
- 先给最精炼的信息（别名 + 定义）→ 让读者快速知道这是什么
- 再给人类深度内容（博主的笔记）→ 核心价值
- 然后给 AI 补充（扩展知识）→ 锦上添花
- 最后是关联内容（引用 + 反向链接）→ 导航到其他相关内容

# Miniese's Blog — 产品需求文档 v0.2

---

## 一、项目概述

### 1.1 定位

Miniese's Blog 是一个面向技术内容创作者的**个人博客 + 知识库系统**，核心特色是内置 AI 助手（Miniese），实现内容发布、知识管理、智能辅助的有机结合。

### 1.2 核心价值

- **对博主**：统一的写作与知识管理平台，AI 辅助降低维护成本，内容持久化且有版本追溯
- **对读者**：流畅的阅读体验，可对文章内容提问，可申请补充知识库词条

### 1.3 命名

- 博客名称：Miniese's Blog
- AI 助手名称：Miniese

---

## 二、用户角色与权限

### 2.1 角色定义

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| 博主 | 唯一管理员 | 全部权限：发布/编辑/删除所有内容，审批 AI 生成内容，管理账户，查看仪表盘 |
| 校内学生 | 通过学校 OAuth 认证的读者 | 可访问标记为"校内可见"的内容，其他同"普通读者" |
| 普通读者 | 未登录或非校内身份 | 仅访问公开内容，可评论，可提交词条申请 |

### 2.2 权限组模型

内容的可见性通过权限组控制：
- 公开：所有人可见
- 校内：仅通过学校 OAuth 认证的学生可见
- 未来可扩展更多权限组

此模型适用于文章、词条和图片。

---

## 三、功能详述

### 3.1 博客文章

#### 3.1.1 内容格式与编译器

- 支持 Markdown 语法
- 支持 Notesaw 语法（block 嵌套结构，向下兼容 Markdown）
- 渲染器设计为可插拔：每个编译器是一个独立模块，接收源文本，输出 HTML
- 博主可在仪表盘添加或切换编译器，以便未来支持更多标记语言

#### 3.1.2 元数据

每篇文章包含以下元数据：
- 标题
- 作者（预留，可能为博主或 AI）
- 发布日期
- 更新日期
- 标签
- 可见性（公开/校内）
- 版权声明（自定义文本 + 可选的开源许可协议）
- 语言（中文/英文）
- Changelog（最近的变更摘要，在文章页特殊渲染）

#### 3.1.3 图片处理

- 渲染时图片宽度缩小至页宽的 60% 左右（保持比例）
- 支持点击灯箱放大查看原图
- 每张图片有独立的可见性设置
- 无权限时显示占位符及提示文字

#### 3.1.4 参考文献

参考文献不做特殊处理，交由 AI 审查时自动检查格式和完整性。博主可在写作时自行维护引用列表。

#### 3.1.5 URL 与多语言

- 文章：`/{语言}/{slug}`
- 词条：`/{语言}/wiki/{词条名}`
- 页面顶部设置全局语言切换按钮，切换时界面文字和内容语言同步切换

#### 3.1.6 文章状态与发布流程

文章分为两种形态：
- **已发布的文章**：对读者可见
- **草稿**：编辑中，不对读者可见

草稿有两种子状态：
- **普通草稿**：可编辑、可直接发布、可提交审查
- **审查中**：已提交 AI 审查，完成后自动回到普通草稿并显示红点提示

**发布流程：**
1. 从草稿点击"发布"，进入发布确认页
2. 上传 Markdown 文件，下方显示渲染预览
3. 展示与上一版本的 diff（源文件对比）及统计信息（增删行数、总长变化）
4. 填写 changelog（AI 生成建议版本，博主可修改）
5. 确认发布 → 文章正式对读者可见，保存版本快照

**审查流程：**
1. 在草稿页点击"提交审查"
2. 草稿变为"审查中"，AI 生成审查报告
3. 报告完成后，草稿状态恢复，显示红点提示
4. 博主查看报告，修改源文件后重新上传
5. 重复或直接发布

每次审查保存快照（文章内容+报告）到本地，便于追溯，但不作为公开版本历史。

#### 3.1.7 评论功能

- **必须登录**：仅登录用户可发表评论
- **频率限制**：同一用户对同一文章 60 秒内最多一条评论
- **AI 初审**：自动检测不当内容（辱骂、广告、垃圾信息等），违规评论自动隐藏并通知博主
- **博主复核**：可在仪表盘将隐藏评论恢复显示，或确认隐藏并自动邮件通知评论者
- 博主也可手动隐藏/显示任意评论

### 3.2 知识库

#### 3.2.1 词条定义与属性

词条是知识库的基本单位，用于解释概念、定理、术语等。

每个词条包含：
- 主名称（标准名称，通常为英文）
- 别名列表（中文译名、简称、常见变体）
- 内容（支持 Markdown/Notesaw）
- 标签
- 可见性（公开/校内）
- 参考文献

#### 3.2.2 多语言策略

中英文词条是**独立的两个词条**，不自动翻译，仅通过链接关联。语言切换时尝试跳转到对应语言版本，如不存在则提示。

#### 3.2.3 双向链接

- **文章 → 词条**：渲染文章时自动检测文本中的词条主名称或别名，添加链接。鼠标悬停时弹出预览框（定义型内容 + "查看完整词条"按钮）
- **词条 → 文章**：词条页面底部自动展示"引用了此词条的文章"列表（反向链接）

#### 3.2.4 AI 词条发现与创建机制

博主发布文章时，AI 自动扫描并生成"新词条提议"列表，发送到仪表盘。提议包含来源（哪篇文章的哪段话）。

博主在仪表盘可：
- 查看每个提议的依据
- 勾选同意创建的词条
- 驳回提议（可注明原因）
- 手动修改词条名称或别名

AI 为每个同意创建的词条分别撰写中文和英文版本，内容包括：
- 主名称与别名列表
- **定义型词条**：简短精准的定义，用于 hover 预览
- **教程型词条**：完整词条页面，包含详细解释、证明、示例、参考文献等

AI 撰写时必须引用外部来源，并在参考文献中列出。AI 生成的词条自动上线，标注"AI生成"，博主后续可在仪表盘校对并移除标注。

#### 3.2.5 词条来源标注

| 来源 | 标注 |
|------|------|
| 人类撰写 | 无额外标注 |
| AI 生成 | 标注"AI生成"，并显示是否经人工审查 |

#### 3.2.6 读者词条申请

读者可选中文章中的一段文本，点击"申请添加词条"按钮直接提交申请。系统自动记录申请位置和选中内容。

防护机制：
- 频率限制（同一用户 5 分钟内最多 3 次）
- AI 初审过滤明显无意义或重复的申请

博主审批时可选择：
- 同意：触发 AI 创建词条
- 驳回：附理由（可选），系统自动邮件通知读者

### 3.3 AI 助手 Miniese

#### 3.3.1 自动触发任务

**发布/更新文章时：**
- 生成 changelog（对比上一版本）
- 生成文章摘要
- 检测关联词条，返回建议列表

**知识库维护：**
- 扫描文章和现有词条，发现新概念则生成词条草稿（中英文各一份），进入仪表盘待审批

#### 3.3.2 手动触发任务

- **扫描旧文章**：在仪表盘触发 AI 扫描指定或全部旧文章，检测可关联词条
- **文章翻译**：手动触发 AI 增量翻译（只翻译修改的段落），生成独立译文文章，标注"AI翻译"

#### 3.3.3 读者交互

**文章对话**：
- 读者可选中文本，向 Miniese 提问
- 助手用自己的知识自由回答，不限文章内容
- 对话结果不对博主可见

**评论初审**：见 3.1.7

#### 3.3.4 AI 行为规范

- 所有 AI 生成内容必须明确标注
- AI 撰写词条时必须引用外部来源
- AI 不自动发布除翻译和词条草稿外的内容（均需博主确认或后续审查）

### 3.4 管理员仪表盘

仪表盘分为四个板块：

#### 3.4.1 文章板块

- 列出所有文章（已发布）及其关联草稿
- 支持按标题、标签、语言、状态筛选和搜索
- 每篇文章显示：标题、语言、最后修改时间、翻译版本状态
- 操作：新建文章、编辑草稿、查看已发布文章、删除文章
- 支持从本地上传 Markdown 文件创建草稿

#### 3.4.2 知识库板块

- **词条管理**：列出所有词条，显示主名称、语言、是否 AI 生成、是否已完成人工审查
- **词条提议审批**：展示来自文章扫描或读者申请的新词条提议，每条包含来源和 AI 初步分析
- AI 撰写的词条草稿自动上线，在此处列出待校对列表，供博主逐条审查并移除标注

#### 3.4.3 通知与活动板块

**通知**：展示需要博主处理的事件：
- 评论被 AI 初审判定违规，等待复核

- 系统提醒（如 token 即将用尽）

每条通知可标记"已处理"或跳转到对应审批页面。

**活动日志**：展示博客动态（博主发布文章、AI 创建词条、自动翻译等），按时间线展示。博主可在仪表盘查看，并可配置是否公开一个独立的"博客更新日志"页面。

通知与邮件发送绑定：重要通知自动发送邮件给博主。

#### 3.4.4 设置板块

- **编译器管理**：查看已安装的编译器，上传新的编译器模块
- **许可协议管理**：配置文章可选的开源许可协议列表
- **Prompt 配置**：各 AI 功能的提示词模板，博主可自定义
- **其他参数**：token 用量监控与警告阈值、自动任务开关等
- **Token 用量监控**：展示近期 AI 调用次数和 token 消耗，超出阈值时邮件警报

- 支持查看历史版本、对比任意两个版本的 diff、回滚到指定版本

## 四、前端设计

### 4.1 整体风格与响应式

- **设计风格**：简洁、现代，圆角卡片为主，适当使用阴影和毛玻璃效果
- **配色**：预定义主题色、强调色等色卡，支持在仪表盘统一调整
- **暗色模式**：支持亮色/暗色/跟随系统切换，用户偏好保存
- **响应式**：移动端适配，手机和平板上有良好阅读体验

### 4.2 页面结构

**全局导航栏**：位于所有页面左侧（移动端折叠为汉堡菜单），包含：
- 主页
- 文章
- 知识库
- 订阅/联系/关于
- 仪表盘（仅博主）
- 登录/个人设置

**页头**：文章页、知识库浏览页、词条页顶部放置头图，包含"Miniese's Blog"字样及简短描述。

### 4.3 页面详细设计

#### 4.3.1 主页

介绍性内容（博客定位、Miniese 简介、近期亮点），可选展示博客更新日志或活动动态摘要。

#### 4.3.2 文章列表页

展示所有已发布文章（时间倒序），支持按标签、语言筛选。每篇文章显示标题、日期、标签、摘要（AI 生成）、语言标识。

#### 4.3.3 文章阅读页

- **正文区域**：居中，默认占页面宽度的 2/3（宽度可在仪表盘设置，读者可在个人设置中覆盖）
- **右侧导航栏**：根据标题自动生成目录，点击跳转，高亮当前位置
- **文末**：评论区、相关词条链接、版权声明、changelog
- **对话入口**：右下角悬浮按钮，点击唤起对话窗口
- **交互**：选中文本后出现浮动工具条，包含"向 Miniese 提问"和"申请添加词条"

#### 4.3.4 词条阅读页

按以下顺序从上到下排列：

1. **标题区**：主名称 + 别名列表（标签形式）
2. **定义型内容区**：一句话解释 + 简短定义，标注"AI生成"（人工校对后移除）
3. **博主撰写的人类内容区**：标题"📝 博主笔记"，无额外标注
4. **AI 撰写的详细内容区**：标题"🤖 AI 补充"，标注"AI生成"及审查状态
5. **文章引用区**（如有）：展示直接讲解此词条的文章段落引用
6. **反向链接区**：列出所有引用了该词条的文章

#### 4.3.5 订阅/联系/关于页面

RSS 链接、邮箱、GitHub 等社交链接、博主介绍、致谢等。

#### 4.3.6 仪表盘

按 3.4 四个板块组织，所有管理操作集中于此。

#### 4.3.7 读者设置页面

- 账户与安全（修改密码、绑定账号）
- 身份认证（学校 OAuth 认证学生身份）
- 偏好设置：界面语言、主题、邮件通知开关、阅读偏好（正文宽度覆盖）

### 4.4 交互细节

#### 4.4.1 文章阅读体验

右侧目录自动生成，滚动时高亮当前位置，点击平滑滚动。

#### 4.4.2 AI 对话窗口（Miniese）

- 右下角固定悬浮按钮，点击从右侧滑出抽屉
- 读者选中文本后，悬浮按钮变为"向 Miniese 提问"，点击后预填选中文本
- 抽屉内：标题、消息记录、输入框、发送按钮
- 移动端：按钮放大，抽屉从底部滑出，占满宽度，键盘弹出时自动上移

#### 4.4.3 词条 hover 预览

鼠标悬停（延迟约 300ms）弹出卡片，显示定义型内容和"查看完整词条"按钮。移动端 tap and hold 触发或直接跳转。

#### 4.4.4 选中文本菜单

选中文章中的文本后出现浮动工具条，包含两个按钮：
- **向 Miniese 提问**：唤起对话窗口并预填文本
- **申请添加词条**：直接提交词条申请（自动填充选中文本作为依据）

移动端简化实现，可只保留提问功能。

#### 4.4.5 暗色模式切换

全局按钮位于导航栏右上角，切换亮色/暗色/跟随系统，保存偏好。

## 五、非功能需求

### 5.1 性能

- 搜索层预留切换到向量数据库的可能

### 5.4 安全性

- OAuth 集成学校统一认证，确保校内内容安全
- 用户密码加密存储
- 评论频率限制防止滥用

# MVP 范围划定 v0.1

- **P2（可以有）**：锦上添花，可无限后延

**MVP = P0 + 少量的 P1（保证体验完整）**，其余 P1 和 P2 留到 v2/v3。

|------|--------|------|
| 支持 Markdown + Notesaw 渲染 | **P0** | 核心能力 |

# 开发工作流设计（为 Vibe Coding 准备）

## 核心原则

1. **契约优先**：先定义接口（API 契约、组件 props、数据库 schema），再实现
2. **垂直切片**：按功能模块垂直切分，而非按技术层（前端/后端/AI）水平切分
3. **可独立验证**：每个模块有明确的输入/输出和测试方式
4. **渐进集成**：先跑通核心链路，再逐步添加边缘功能

---

## 推荐工作流

```
Step 1: 写接口契约（Contract）
        ↓
Step 2: 生成 Mock 实现 + 测试用例
        ↓
Step 3: 逐模块实现（后端 → AI → 前端）
        ↓
Step 4: 集成测试（替换 Mock 为真实实现）
        ↓
Step 5: 端到端验证
```

## Step 1：定义接口契约

每个模块对外暴露清晰的接口，不依赖内部实现细节。

**示例：文章发布接口**

```typescript
// 契约：POST /api/articles/publish
// 输入
interface PublishRequest {
  file: File;           // Markdown 文件
  changelog: string;    // 用户填写的 changelog
}

// 输出
interface PublishResponse {
  articleId: string;
  slug: string;
  url: string;
  reviewReport?: ReviewReport;  // 如果触发了审查
}
```

**契约文档包含**：
- 接口路径、方法、请求/响应格式
- 错误码
- 边界条件（如文件过大、格式错误）

这样前端和后端可以并行开发：前端用 Mock 数据，后端按契约实现。

## 1. 总体架构

系统采用 Next.js 全栈架构，附加独立的队列 Worker 处理长时间 AI 任务。

### 1.1 组成部分

| 组件 | 职责 | 运行方式 |
|------|------|----------|
| Next.js 应用 | 页面渲染 + API Routes（短任务） | 主进程 |
| Redis | 队列存储（Bull） | 独立容器 |
| Worker | 消费队列中的长任务（AI 审查、翻译、词条生成） | 独立进程 |
| PostgreSQL | 数据存储 | 独立容器 |

### 1.2 通信流程

- 用户请求 → Next.js API Route → 创建任务存入 Redis 队列 → 立即返回任务 ID
- Worker 从队列取任务 → 调用 DeepSeek API → 更新数据库中的任务结果
- 前端轮询任务状态 API → 获取结果后展示

### 1.3 短任务 vs 长任务

| 类型 | 直接处理 | 走队列 |
|------|----------|--------|
| 文章发布（不含审查） | ✅ | |
| 读取文章列表 | ✅ | |
| 读者对话 | ✅ | |
| 发送邮件 | ✅ | |
| AI 审查 | | ✅ |
| AI 翻译 | | ✅ |
| AI 词条生成 | | ✅ |
| 扫描旧文章 | | ✅ |

## 5. 队列设计

### 5.1 任务类型

| 类型 | 输入 | 输出 |
|------|------|------|
| review | articleId, 文章内容 | 审查报告（结构化 JSON） |
| translate | articleId, 源语言, 目标语言 | 译文 Markdown 文件路径 |
| generate | wikiEntryProposalId | 中英文词条内容 |
| scan | （可选）文章范围 | 新词条提议列表 |

### 5.2 任务生命周期

1. API 调用 `producer.add(jobType, data)` → 返回 `job.id`
2. Worker 处理：更新数据库状态为 `processing` → 调用 AI → 更新结果
3. 前端轮询 `/api/ai/status?id=xxx` → 拿到结果后展示

### 5.3 Worker 实现要点

- 单独进程，`node dist/worker.js`
- 监听 `completed` 和 `failed` 事件，更新数据库
- 失败重试机制（最多 3 次）

## 6. AI 集成设计

### 6.1 DeepSeek API 调用封装

```typescript
// lib/ai/client.ts
async function callDeepSeek(prompt: string, responseFormat?: "json"): Promise<string>
```

- 统一错误处理（重试、超时）
- 支持 JSON 模式（要求 AI 返回合法 JSON）
- 记录每次调用的 token 消耗（用于监控）

### 6.2 提示词管理

- 存储在 `lib/ai/prompts/` 目录下，每个功能一个 `.txt` 或 `.ts` 文件
- 支持变量插值（如 `{{articleContent}}`）
- 博主可在仪表盘修改（未来功能）

## 7. Markdown 渲染设计

### 7.1 渲染流程

1. 读取 MD 文件
2. 使用 unified 生态（remark/rehype）解析
3. 集成 Notesaw 插件（处理 block 语法）
4. 集成 remark-math（KaTeX）
5. 输出 HTML

### 7.2 词条链接检测

- 渲染前扫描 MD 内容，匹配词条主名称和别名
- 将匹配的文本替换为 `<a href="/wiki/...">` 标签
- hover 数据（定义型内容）通过 `data-wiki` 属性存储，前端 JS 处理预览

### 7.3 渲染时机

- **发布时渲染**：生成 HTML 存入数据库或缓存文件
- 理由：文章更新频率低，避免每次请求解析

## 3. 开发流程

### 3.1 标准工作流

```
1. 阅读相关文档
   │
2. 编写测试用例（单元测试/集成测试）
   │
3. 实现功能代码
   │
4. 运行测试，确保通过
   │
5. 更新相关文档（如有变更）
   │
6. 更新 development-log.md
   │
7. 提交代码（git commit）
   │
8. 汇报完成
```

### 3.2 代码规范

| 项目 | 要求 |
|------|------|
| 语言 | TypeScript，严格模式（`strict: true`） |
| 命名 | 变量/函数：camelCase；类/接口：PascalCase；常量：UPPER_SNAKE_CASE |
| 注释 | 公共函数必须有 JSDoc 注释 |
| 类型 | 禁止使用 `any`，必要时用 `unknown` |
| 错误处理 | 使用 `try/catch`，记录错误日志 |
| 异步 | 优先使用 `async/await` |

### 3.3 测试要求

| 类型 | 覆盖率要求 | 说明 |
|------|------------|------|
| 单元测试 | ≥ 80% | 测试纯函数、工具函数 |
| 集成测试 | 核心 API 全部覆盖 | 使用 Supertest + 测试数据库 |
| E2E 测试 | 只测核心路径（可选） | 使用 Playwright |

**测试文件位置**：
- 单元测试：与源文件同目录，`*.test.ts`
- 集成测试：`tests/integration/`
- E2E 测试：`tests/e2e/`

### 3.4 文档更新要求

| 变更类型 | 需更新的文档 |
|----------|--------------|
| 功能逻辑变更 | `docs/PRD.md`（如影响需求） |
| 技术实现变更 | `docs/architecture.md` |
| 数据库 schema 变更 | `prisma/schema.prisma` + `docs/architecture.md` |
| API 接口变更 | `docs/contracts/*.ts`（如有） |
| 任务顺序/依赖变更 | `docs/development-order.md` |
| 所有变更 | `docs/development-log.md` + `CHANGELOG.md` |

### 3.5 Git 提交规范

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

**type**：
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `test`: 测试相关
- `refactor`: 重构
- `chore`: 构建/工具配置

**示例**：`feat(article): add publish API with review option`
